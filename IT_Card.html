01. Infrastructure Architecture

물리적 인프라(On-Premise)와
가상화 서비스(Cloud)

Definition: On-Premise vs Cloud

온프레미스는 조직이 물리적으로 소유한 공간에 직접 서버 장비를 구축·운영하는 방식이며, 클라우드는 외부 공급자(AWS, Azure 등)의 가상화된 컴퓨팅 자원을 네트워크를 통해 사용한 만큼 비용을 지불하고 이용하는 서비스 모델입니다.

과거 시스템 구축의 정석이었던 온프레미스는 보안성이 뛰어나지만 서버 도입까지 수개월의 조달 기간이 소요되며, 트래픽 폭주 시 즉각적인 대응이 어렵습니다.

반면, 클라우드는 클릭 몇 번으로 서버 사양을 조절(Auto-scaling)할 수 있어 LG전자의 글로벌 마케팅 이벤트나 신규 서비스 런칭 시 발생하는 불확실한 접속량에 유연하게 대응할 수 있습니다. IT 파트너가 "클라우드 네이티브"를 언급한다면, 시스템 설계 단계부터 클라우드의 이러한 가변성과 고가용성을 최우선으로 고려하자는 전략적 제안입니다.

02. Network Communication

상호작용의 시작과 끝,
클라이언트(Client)와 서버(Server)

Definition: Client & Server

클라이언트는 사용자에게 정보 출력을 요청하고 결과를 표시하는 인터페이스(앱/웹)를 의미하며, 서버는 네트워크를 통해 클라이언트로부터 수신된 요청을 처리하고 데이터를 가공하여 응답을 반환하는 고성능 컴퓨터 시스템입니다.

우리가 일상적으로 사용하는 LG ThinQ 앱의 화면 구성이나 버튼들은 모두 클라이언트 사이드에 속합니다. 사용자가 "건조기 상태 확인" 버튼을 누르는 행위는 서버를 향한 '요청(Request)'이 됩니다.

서버는 해당 제품의 실시간 상태 값을 DB에서 조회하여 다시 앱으로 전송(Response)합니다. 만약 "서버 응답이 없다"는 보고를 받는다면, 이는 주방(서버)에서 주문(요청)을 처리하지 못할 만큼 과부하가 걸렸거나 통신 경로에 문제가 생겼음을 의미합니다. 기획 시 사용자 경험(UX)뿐만 아니라 서버 부하량까지 고려해야 하는 이유입니다.

04. Interface & Integration

시스템 간 소통의 약속,
API (응용 프로그램 인터페이스)

Definition: API

애플리케이션 간에 데이터와 기능을 서로 주고받기 위해 정의된 통신 규격입니다. 상호 작용을 위한 '창구'이자 소프트웨어 컴포넌트 간의 '계약'으로 정의됩니다.

우리의 시스템이 외부 결제 서비스나 배송 추적 서비스와 연동될 때, 상대방의 소스 코드를 직접 가져오는 것이 아니라 그들이 열어둔 API 창구를 이용합니다.

IT 부서에서 "API 명세서를 확인해야 한다"고 말한다면, 그것은 어떤 데이터 형식을 보내고 받아야 하는지 기술된 가이드(규약)를 체크하겠다는 뜻입니다. 현업에서는 신규 솔루션 도입 시 "우리 내부 시스템과 API 연동이 자유로운가?"를 체크하는 것이 향후 확장성 확보에 필수적입니다.

05. Data Interchange Format

데이터를 담는 표준 양식,
JSON (제이슨)

Definition: JSON

JavaScript Object Notation의 약자로, 속성-값 쌍(Attribute-Value Pairs)으로 구성된 텍스트 기반의 데이터 교환 포맷입니다. 기계가 파싱하기 쉽고 인간이 읽기 편한 구조를 가지고 있습니다.

시스템 간 대화 시 {"제품": "TV", "가격": 2500000}와 같은 형태를 사용합니다. 과거의 복잡했던 XML 포맷보다 가볍고 직관적이어서 현재 전 세계적으로 표준처럼 쓰입니다.

기획자가 화면에 들어갈 데이터 항목을 정의할 때, IT 담당자와 이 JSON 구조(Key와 Value의 관계)에 대해 논의하면 데이터 누락 없는 정확한 기능 구현이 가능해집니다.

08. Server Logic

서버의 두 가지 심장,
Web Server와 WAS

Definition: Web Server vs WAS

웹 서버는 정적 콘텐츠(HTML, 이미지)를 제공하며, WAS는 비즈니스 로직을 수행하고 DB 조회를 거쳐 동적 데이터를 생성하여 제공합니다.

똑같이 보이는 홈페이지 대문 이미지는 Web Server가 빠르게 전달합니다. 하지만 로그인을 하거나 제품 추천 리스트를 가져오는 것은 WAS가 내부 로직을 계산해서 보내줘야 합니다.

시스템 성능 지연 보고 시 "WAS 리소스 점검"이라는 표현이 나온다면, 복잡한 비즈니스 계산이나 DB 접근 과정에서 병목 현상이 발생했다는 뜻입니다.

09. Database Query Language

데이터를 뽑아내는 마법의 문장,
SQL (구조화 질의 언어)

Definition: SQL

관계형 데이터베이스(RDBMS)에서 데이터를 관리하고 조작하기 위해 설계된 특수 목적의 프로그래밍 언어입니다.

기획자가 IT에 데이터 추출 요청을 할 때 가장 많이 언급되는 단어입니다. "SQL을 짠다"는 것은 거대한 데이터베이스 표들 사이에서 내가 원하는 조건만 필터링하여 가져오는 명령문을 작성하는 것입니다.

기초적인 SQL 구문을 이해하고 있으면 IT 부서에 의존하지 않고도 직접 분석용 저장소(DW)에서 필요한 통계치를 추출하여 신속한 의사결정을 내릴 수 있습니다.

10. Infrastructure Virtualization

어디서든 똑같이 돌아가는 패키지,
도커(Docker)와 컨테이너

Definition: Docker

애플리케이션과 환경 설정을 하나의 '컨테이너'에 담아 실행 환경의 독립성을 보장하는 플랫폼입니다.

개발자의 PC에서는 잘 돌아가는데 운영 서버에만 올리면 에러가 나는 문제는 서버마다 설정이 다르기 때문입니다. 도커는 모든 환경을 하나의 가방(컨테이너)에 통째로 담습니다.

이 가방만 옮기면 어디서든 똑같은 동작을 보장하므로 배포 효율이 비약적으로 향상됩니다. 현대적인 클라우드 기반 프로젝트에서 인프라 구성을 논의할 때 빠지지 않는 핵심 기술입니다.

11. Release Management

코드 완료가 끝이 아니다,
배포(Deploy)와 자동화(CI/CD)

Definition: Deployment & CI/CD

배포는 소프트웨어를 실제 운영 환경에 설치하여 사용자에게 제공하는 과정이며, CI/CD는 이 과정을 자동화하는 파이프라인 기술입니다.

배포 자동화(CI/CD)가 잘 구축된 조직은 하루에도 여러 번 작은 기능들을 안전하게 서비스에 반영할 수 있습니다. 이는 고객 피드백에 기민하게 반응해야 하는 프로젝트에 매우 유리한 기술적 인프라입니다.

12. High Availability

트래픽 정체를 해결하는 교통 정리,
로드밸런싱(Load Balancing)

Definition: Load Balancing

서버에 들어오는 요청을 여러 대의 서버에 균등하게 분산시켜주는 장치나 기술입니다.

갑작스러운 신제품 알림톡을 보내면 수십만 명의 고객이 한꺼번에 접속합니다. 이때 로드밸런서는 교통경찰처럼 요청들을 놀고 있는 서버들로 적절히 나누어 보냅니다.

대규모 이벤트 시의 서비스 장애를 사전에 방지하기 위해 로드밸런싱 설정이 충분한지 IT에 미리 질의하는 습관이 중요합니다.

07. AI Evolution

알파고에서 챗GPT까지,
AI · ML · DL의 핵심 개념

Definition: AI > ML > DL

AI(인공지능)는 인간의 지능을 모사하는 가장 넓은 개념이며, ML(머신러닝)은 데이터로부터 기계가 스스로 학습하는 방법론, DL(딥러닝)은 인간의 뇌 구조를 본뜬 인공신경망을 통해 복잡한 패턴을 찾아내는 기술입니다.

우리는 2016년 '알파고(AlphaGo)'를 통해 딥러닝의 위력을 실감했습니다. 특정 규칙(바둑) 내에서 최적의 수를 찾는 '전문가형 AI' 시대였습니다.

하지만 2020년 GPT-3.0이 발표되면서 판도가 바뀌었습니다. 특정 목적이 아닌, 방대한 데이터를 학습하여 '인간처럼 대화하고 창작하는' 생성형 AI(Generative AI)의 시대가 열린 것입니다.

Artificial Intelligence (인공지능)
Machine Learning (기계학습)
Deep Learning (심층학습)

※ 하위 개념으로 갈수록 데이터 집약적이며 복잡한 처리가 가능합니다.

왜 LLM(거대언어모델)이 좋은가요?

과거 AI는 사진 한 장을 분류하기 위해 수만 장의 사진을 새로 학습시켜야 했습니다. 하지만 LLM은 이미 세상의 거의 모든 지식을 학습했기에, 아주 적은 힌트(프롬프트)만으로도 우리 비즈니스에 맞는 결과를 즉시 만들어낼 수 있습니다. 이것이 혁신의 속도를 바꾸고 있습니다.

13. Generative AI

인간처럼 이해하고 생성하는,
거대 언어 모델(LLM)

Definition: LLM

방대한 텍스트 데이터를 학습하여 자연어 이해 및 생성 능력을 갖춘 딥러닝 모델입니다.

최근 LG전자에서 '공감 지능'을 구현하기 위한 핵심 도구입니다. 고객 상담 데이터를 요약하거나, 사용자의 명령을 가전 제품이 알아듣도록 번역하는 데 사용됩니다.

단, LLM은 아는 척하며 오답을 말하는 '할루시네이션(환각)' 현상이 있습니다. 이를 보완하기 위해 RAG 기술을 결합하여 정확도를 높이는 것이 기획 포인트입니다.

14. Knowledge-Based AI

검색 증강 생성(RAG)

Definition: RAG

AI가 답변 전, 신뢰할 수 있는 외부 지식(매뉴얼 등)에서 정보를 '검색'한 뒤 답변을 구성하게 하는 기술입니다.

범용적인 AI는 LG전자의 최신 매뉴얼을 알지 못합니다. RAG를 적용하면 답변 직전에 "최신 OLED TV 매뉴얼"을 뒤져보고 그 근거를 바탕으로 대답하여 신뢰도를 높입니다.
