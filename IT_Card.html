04. Interface & Real-time Integration

시스템 연동의 실시간 심장,
API (Application Programming Interface)

서로 다른 시스템이 약속된 URL 주소를 통해 실시간으로 데이터를 주고받는 연결 창구입니다.

왜 DB-to-DB가 아닌 API인가?

우리는 보통 데이터베이스끼리 직접 연결하는 방식을 떠올리지만, 실시간성이 중요한 현대 IT에서는 API 송/수신 방식을 활용합니다. 특정 정보를 원할 때 URL 주소(Endpoint)를 호출하여 즉시 답을 받기 때문입니다.

GET https://api.lg-service.com/v1/customer/status

CS 핵심 사례: 고객 OneView

상담사가 고객 OneView를 열 때, 물류 시스템의 배송 현황이나 IoT 상태 정보를 실시간 API 호출로 가져옵니다. DB가 동기화되길 기다리지 않고 즉시 데이터를 수신하는 것이 핵심입니다.

05. Data Interchange Format

데이터를 담는 글로벌 표준 바구니,
JSON (제이슨)

Abbreviation: JSON

JavaScript Object Notation의 약자로, 텍스트 기반의 경량 데이터 교환 포맷입니다.

"사람 이름이 아닙니다"

옆자리 제이슨 선임님이나 공포 영화 주인공이 아닙니다. 과거 무거운 XML보다 사람이 읽기 쉽고 가벼워서 현재 전 세계 모든 시스템의 '공용어'로 선택받았습니다.

왜 '비정형' 데이터의 최강자인가?

"기존 테이블 구조(RDBMS)의 애로사항을 해결합니다"

  • 기존 테이블: 열 하나만 추가해도 전체 DB 구조를 뜯어고쳐야 하며(Schema change), 시스템이 중단될 위험이 큽니다.
  • JSON의 유연함: 고정된 칸이 없습니다. 오늘은 '고객명'만 보내고, 내일은 '상담메모'와 'IoT로그'를 갑자기 추가해도 시스템이 깨지지 않습니다.
  • 이런 '틀에 얽매이지 않는 구조' 때문에 비정형/반정형 데이터를 실시간으로 처리하기에 최적입니다.
Real-world JSON Log
{
  "고객ID": "LG-C001",
  "증상": "전원 불량",
  "IoT_Logs": {
    "volt": 220,
    "error_code": "E61"
  },
  // 구조 변경 없이 내일 즉시 추가 가능
  "내일_추가될_필드": "상담원 특이사항"
}

06. Development Roles

가치를 만드는 4대 핵심 역할,
디자인부터 백엔드까지

Definition: UX/UI to Backend Flow

하나의 IT 제품은 사용자의 눈에 보이는 디자인(Design), 이를 브라우저 언어로 옮기는 퍼블리싱(Publishing), 화면의 기능을 구현하는 프론트엔드(Frontend), 데이터와 로직을 처리하는 백엔드(Backend)의 유기적 협업으로 완성됩니다.

🎨
디자이너 (Designer)
"어떻게 보일까?"

사용자 경험(UX)을 설계하고 시각적 요소(UI) 가이드를 제작합니다. 상담원의 시선 이동과 편의성을 최우선으로 고려합니다.

📑
퍼블리셔 (Publisher)
"뼈대를 만들까?"

디자인 시안을 웹 표준 언어(HTML/CSS)로 변환합니다. 모든 브라우저에서 화면이 깨지지 않고 똑같이 보이게 정렬합니다.

💻
프론트엔드 (Frontend)
"어떻게 동작할까?"

사용자 동작과 데이터를 연결합니다. 버튼 클릭 시 팝업을 띄우거나, API 정보를 화면에 실시간으로 뿌려주는 역할을 합니다.

⚙️
백엔드 (Backend)
"어떻게 처리할까?"

데이터 처리와 핵심 비즈니스 로직을 담당합니다. DB 재고 조회나 상담 저장을 처리하는 '보이지 않는 엔진'입니다.

💡 기획자와의 소통 Point:

"디자인 가이드와 달라요"
→ 디자이너/퍼블리셔 소통
"버튼 클릭 시 화면이 안 변해요"
→ 프론트엔드 소통
"데이터 저장이 안 돼요"
→ 백엔드 소통

07. Server Logic & Performance

서버의 두 가지 심장,
Web Server와 WAS

Core Concepts

웹 서버는 변하지 않는 자료를 전달하고, WAS는 상황에 맞춰 정보를 가공하여 전달합니다.

Web Server (웹 서버)

정적 콘텐츠(Static)를 처리합니다. 누구나 똑같이 보는 파일을 '빠르게' 꺼내주는 매장 카운터와 같습니다.

사용 상황
  • 제품 이미지(JPG/PNG) 로딩
  • 서비스 매뉴얼 PDF 파일 다운로드
  • 화면 구성용 고정 스타일(CSS) 파일 전달

WAS (웹 어플리케이션 서버)

동적 콘텐츠(Dynamic)를 처리합니다. 사용자의 요청에 따라 주방에서 '요리'를 하듯 로직을 수행하는 핵심 장치입니다.

사용 상황
  • 실시간 수리 비용 계산 및 견적 조회
  • 고객별 맞춤 상담 이력 DB 호출
  • API를 통한 실시간 배송 현황 추적

💡 왜 두 서버를 나누나요? (Separation of Concerns)

모든 일을 하나의 서버(WAS)가 다 하면 부하가 걸려 시스템이 멈춥니다. 이미지 같은 가벼운 일은 웹 서버가 입구에서 처리하고, 복잡한 계산은 WAS가 전담하게 하여 시스템의 효율과 안정성을 극대화합니다.

08. Database & SQL

데이터를 뽑아내는 마법의 문장,
SQL (구조화 질의 언어)

Definition: SQL

관계형 데이터베이스(RDBMS)에서 데이터를 관리하고 조작하기 위해 설계된 특수 목적의 프로그래밍 언어입니다.

"SQL을 짠다"는 것은 거대한 데이터베이스 표들 사이에서 내가 원하는 조건만 필터링하여 가져오는 명령문을 작성하는 것입니다. 기초적인 SQL을 이해하면 IT 부서의 도움 없이도 직접 DW(데이터 웨어하우스)에서 필요한 통계치를 추출할 수 있습니다.

RDBMS (관계형)

  • • 구조: 엑셀처럼 정해진 표(Table) 형식에 데이터 저장
  • • 특징: 데이터의 정확성과 일관성이 매우 중요함
  • • 예시: Smile Plus 상담 접수, 부품 재고 관리, 결제 정보
  • • 언어: SQL을 사용하여 데이터를 관리

NoSQL (비관계형)

  • • 구조: 고정된 형식 없이 유연한 문서(JSON 등) 형태로 저장
  • • 특징: 대량의 데이터를 빠르게 저장하고 확장하기 유리함
  • • 예시: 가전 IoT 센서 로그, 대규모 메신저 대화 이력, 캐시 데이터
  • • 언어: SQL이 아닌 고유의 방식으로 데이터를 관리
SELECT voc_text FROM service_logs WHERE model='L-Wash-01';

09. Basic IT Glossary

협업의 해상도를 높이는
필수 기본 IT 용어

시스템 연계 및 인터페이스

서로 다른 시스템이 데이터를 주고받을 수 있도록 '연결'하는 행위 자체를 시스템 연계라고 하며, 그 연결 통로나 규격을 인터페이스(Interface)라고 부릅니다.

배치 (Batch)

실시간으로 즉시 처리하는 것이 아니라, 대량의 데이터를 정해진 시간(예: 매일 새벽 2시)에 일괄적으로 처리하는 방식입니다. 주로 대용량 통계 생성 시 활용됩니다.

시스템 간의 대화: Request & Return

리퀘스트 (Request)

클라이언트가 서버에게 "이 데이터 줘"라고 던지는 질문/요청입니다.

리턴 (Return/Response)

서버가 요청을 처리한 후 다시 보내주는 답변/결과값입니다.

파라미터 (Parameter)

요청할 때 함께 보내는 세부 조건입니다. (예: '고객명'이라는 질문에 '홍길동'이라는 파라미터를 담아 보냄)

💡 "오늘자 VOC 데이터를 배치로 돌려서 마트에 적재한 뒤, 인터페이스 파라미터 값을 확인해 봅시다."
이제 이 문장이 자연스럽게 들린다면 여러분은 이미 IT 파트너와 소통할 준비가 되었습니다!

13. AI Optimization Strategy

AI를 효율적으로 활용하는 방법:
성능 최적화 3대 핵심 기술

프롬프트 엔지니어링

핵심 개념: 모델의 파라미터를 수정하지 않고, 질문(Prompt)의 구성만 최적화하여 결과 품질을 높이는 기술입니다.

장점
즉각 적용 가능, 비용 거의 없음
단점
일관성 부족, 복잡한 로직 한계

"인턴에게 상세한 업무 지시서를 주는 것과 같습니다."

RAG (검색 증강 생성)

핵심 개념: 답변 생성 전, 외부 지식베이스에서 신뢰할 수 있는 정보를 먼저 검색한 뒤 이를 참고하여 답변을 만듭니다.

장점
환각 방지, 최신 정보 즉시 반영
단점
검색 속도 지연, 구축 비용 발생

"AI에게 오픈북 테스트(사전 지참)를 시키는 것입니다."

파인튜닝 (미세 조정)

핵심 개념: 특정 목적의 데이터를 사용하여 AI 모델 자체를 추가 학습시켜 모델의 가중치(Parameter)를 수정합니다.

장점
고유 말투/지식 완벽 내재화
단점
고비용, 데이터 업데이트 어려움

"범용 직원을 특수 분야 전문가로 양성하는 교육입니다."

결론: 프롬프트는 기본 도구이며, 데이터 신뢰도가 중요하다면 RAG를,
기업 고유의 정체성이나 고도의 전문성이 필요하다면 파인튜닝을 결합하여 최적의 성능을 냅니다.

11. Modern AI Narrative

지능에서 감각, 그리고 자율적인 행동으로:
현대 AI의 진화 스토리

Stage 1. 지능

Foundation Model

거대한 '지식의 뇌'를 구축하는 기초 단계입니다.

장점: 한 번의 학습으로 요약, 코딩 등 범용적 활용 가능

단점: 천문학적 비용, 할루시네이션(환각) 리스크

Stage 2. 감각

Multi-modal AI

뇌에 '보고 듣는 오감'이 연결되는 단계입니다.

장점: 사진/음성 이해로 인간과 더 풍부한 소통 가능

단점: 데이터 복잡도 증가로 인한 연산 처리 지연

Stage 3. 자율

AI Agent

지능을 이용해 '스스로 판단하고 실행'하는 단계입니다.

장점: 목표만 주어지면 과정을 스스로 계획하고 업무 처리

단점: 자율적인 판단에 따른 제어 불능 및 보안 이슈

Stage 4. 행동

Physical AI

자율 지능이 '현실의 신체(기기)'와 결합하는 최종 단계입니다.

장점: 현실 세계에서 물리적인 도움(로봇, 가전) 제공

단점: 물리적 충돌 등 안전성과 고도의 신뢰성 요구

"뇌(Foundation)와 오감(Multi-modal)을 가진 AI가 스스로 계획(Agent)하고
마침내 신체(Physical)를 얻어 현실을 움직이는 존재로 거듭나고 있습니다."

12. Sovereign AI

데이터와 기술의 독립:
왜 지금 소버린 AI(Sovereign AI)인가?

Sovereign AI의 필요성

  • • 데이터 주권: 국가/기업의 핵심 정보 유출 방지 및 보안 강화
  • • 문화적 이해: 자국어의 뉘앙스와 특유의 정서를 정확히 반영
  • • 기술 독립: 외산 모델의 정책 변화나 비용 인상에 휘둘리지 않는 자립

대한민국 정부의 전략

우리 정부는 'AI 주권 확보'를 국가 과제로 삼고 있습니다. 한국형 거대 모델 개발 예산 투입 및 공공 데이터 개방을 통해 독자적인 AI 생태계 구축을 적극 지원하고 있습니다.
