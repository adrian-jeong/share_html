Network Communication

서비스를 주고받는 주체:
클라이언트(Client)와 서버(Server)

클라이언트는 서비스를 요청하는 창구이며, 서버는 24시간 대기하며 요청을 처리하고 응답을 반환하는 핵심 시스템입니다.

카카오톡의 원리

사용자의 폰에서 돌아가는 카카오톡 앱은 메시지 전송을 요청하는 '클라이언트'입니다. 수억 건의 메시지를 받아 친구에게 정확히 전달하고 보관하는 카카오 본사의 거대 컴퓨터가 바로 '서버'입니다.

어떻게 찾아갈까? (IP & Port)

IP (Internet Protocol) : "집 주소"

네트워크 상에서 서버의 물리적인 위치입니다. 예: 172.217.161.132 (전 세계 유일)

Port (포트) : "방 번호 / 출입문"

서버 한 대에는 여러 서비스가 있습니다. 웹은 80번, 메일은 25번처럼 어떤 창구로 들어갈지 결정하는 번호입니다.

메시지 보내줘!

요청(Request)

응답(Response)

Port: 443

Infrastructure Architecture

인프라의 거대한 변화:
온프레미스(On-Premise) vs 클라우드(Cloud)

온프레미스는 서버를 직접 사서 내 건물 지하에 두는 방식이고, 클라우드는 AWS나 구글 같은 거대 인프라를 빌려 쓰는 방식입니다.

내 집 마련 vs 호텔 숙박

온프레미스 (내 집 마련) 부지를 사고 집을 직접 짓습니다. 구조를 내 맘대로 통제하지만, 수도가 고장 나면 내가 고쳐야 하고 식구가 늘어나 이사하려면 수개월이 걸립니다.
클라우드 (호텔 숙박) 이미 지어진 방을 빌립니다. 청소와 관리는 호텔(AWS 등)이 다 해줍니다. 손님이 갑자기 몰리면 바로 옆방을 더 빌리면 됩니다. 대신 체크아웃 때 빌린 만큼 요금을 냅니다.
!

💡 온프레미스의 리스크: "비둘기의 습격"

스타트업 근무 시절, 서버실 에어컨 실외기에 비둘기가 집을 짓는 바람에 냉방이 마비된 적이 있었습니다. 서버실이 대표님 방 뒤편이었는데, 급한 대로 대표님 방 에어컨을 풀 가동하고 대형 선풍기 두 대로 공기를 억지로 밀어 넣었죠.

하지만 서버는 열기를 버티지 못하고 뻗어버렸습니다. 하필 K은행 프로젝트 솔루션 업그레이드 마감이 코앞인 상황! 결국 고사양 워크스테이션을 가진 수석님 자리에 개발자 여러 명이 옹기종기 달라붙어 밤샘 작업을 하며 위기를 넘겼습니다.

Data Architecture & Pipeline

데이터 생애주기별 저장소 체계:
DB, DL, DW, DM 핵심 요약

Operation (운영)

DB (RDBMS)

  • 성격: 실시간 트랜잭션 처리(OLTP)
  • 데이터: 고도로 정제된 정형 데이터
  • 목적: 서비스 운영 및 데이터 무결성
  • 예시: 고객 프로필, 실시간 주문/재고
Raw Storage (원천)

DL (Object)

  • 성격: 가공되지 않은 원천 데이터 보관
  • 데이터: 정형/반정형/비정형(로그, 이미지)
  • 목적: 데이터 유실 방지 및 AI 학습용
  • 예시: 가전 IoT 센서 로그, VOC 녹취록
Integrated (통합)

DW (Analytics)

  • 성격: 전사적 통합 분석(OLAP)
  • 데이터: 여러 소스의 통합/정제/이력화
  • 목적: Single Version of Truth 확보
  • 예시: 전사 판매 실적, 고객 생애 가치(LTV)
End-User (활용)

DM (Subject)

  • 성격: 특정 주제/부서별 추출 데이터
  • 데이터: 고도로 요약/집계된 데이터셋
  • 목적: 빠른 조회 및 현업 활용 편의
  • 예시: 지역별 수리 효율, 마케팅 타겟팅

기술적 팩트 체크

• ETL 프로세스: 데이터를 추출(Extract), 변환(Transform), 적재(Load)하는 과정이 저장소 사이를 연결합니다.

• 데이터 거버넌스: 팩트 중심 분석을 위해선 각 저장소 간 데이터 정합성을 맞추는 관리가 필수적입니다.

현업 실무 포인트: CS 데이터마트의 성격

우리가 현재 운영 중인 CS 데이터마트는 명칭상 '마트(DM)'로 분류되지만, 실제로는 글로벌 서비스 로그와 방대한 고객 데이터를 체계적으로 통합/정제한 DW(데이터 웨어하우스)의 성격을 띄고 있습니다.

단순한 부서용 요약 데이터셋을 넘어, 전사 CS 분석의 'Standard Master' 역할을 수행하기 때문에 고도의 데이터 정합성이 요구되는 저장소입니다.

Understanding API (Application Programming Interface)

실시간 데이터의 디지털 통로:
API란 무엇인가?

"API는 서로 다른 시스템이 실시간으로 데이터를 주고받기 위한 약속입니다."
현업의 요구사항이 '즉각적인 정보 확인'이라면, 기술적 해답은 반드시 API여야 합니다.

실시간 API의 역할

REAL-TIME
활용 예: 실시간 현황 대시보드

고객센터의 상담 대기 전광판이나 실시간 수리 현황판처럼 '지금 바로'의 상태가 중요한 화면에 사용합니다.

Data Exchange Format Example
GET /v1/delivery-status
{ "order_id": 1234, "status": "In Transit" }
※ 상세한 데이터 형식(JSON)은 다음 페이지에서 다룹니다.

데이터 배치(Batch) 작업

SCHEDULED
활용 예: 전일 마감 실적 통계

어제까지의 정산 데이터나 전일 판매 실적 리포트처럼 양이 방대하고 즉각적인 업데이트가 필요 없는 경우에 효율적입니다.

DB to DB 데이터 연결은 주로 이 방식으로 이루어지며, 전송 주기가 정해져 있어 현시점 데이터와의 딜레이가 필연적으로 발생합니다.

배치(Batch): 데이터를 실시간으로 전송하지 않고, 정해진 주기(예: 매일 새벽)마다 한꺼번에 대량으로 처리하는 방식입니다.

참고 (REST API): 현재 전 세계적으로 가장 널리 쓰이는 API 설계 표준입니다. 웹 브라우저의 주소창과 같은 규칙(HTTP)을 사용하기 때문에 매우 직관적이고 소통이 간결합니다.

SQL (구조화 질의 언어)

Core Concept

SQL은 관계형 데이터베이스(RDBMS)에 저장된 거대한 데이터 뭉치 속에서
내가 원하는 정보만 '필터링'하여 가져오기 위한 비즈니스 공용어입니다.

RDBMS (관계형 DB)

STRUCTURED

엑셀처럼 행과 열이 정해진 표 형식에 데이터를 엄격히 관리합니다. 데이터의 정확성이 생명입니다.

  • 특징: 중복 데이터 방지, 고도의 정합성 유지
  • 주요 데이터: Smile Plus 상담 이력, 부품 재고 관리, 결제 정보 등
  • 언어: 오직 SQL로만 대화 가능

NoSQL (비관계형 DB)

UNSTRUCTURED

고정된 형식 없이 유연한 문서(JSON 등) 형태로 저장합니다. 속도와 확장성이 생명입니다.

  • 특징: 복잡한 규칙 없이 대량의 데이터 고속 저장
  • 주요 데이터: 가전 IoT 센서 로그, 대규모 메신저 대화, 캐시 등
  • 언어: SQL이 아닌 고유의 방식 (속도 중심)

Basic IT Glossary

협업의 해상도를 높이는
실무 핵심 IT 용어 사전

데이터 처리 방식 (OLTP/OLAP)

OLTP (실시간 트랜잭션)

"현재의 거래를 기록하는 엔진". 무결성이 핵심이며, 주문 결제나 상담 접수처럼 실시간 데이터 수정/입력에 최적화되어 있습니다.

OLAP (분석용 프로세싱)

"과거 데이터를 분석하는 창고". 대량 데이터를 다각도에서 조회하고 통계를 내는 데 최적화되어 의사결정을 돕습니다.

프로토콜 (Protocol)

시스템이나 기기 간에 데이터를 주고받기 위해 정한 "대화 예절(통신 규약)"입니다. HTTP, HTTPS 등이 대표적인 프로토콜입니다.

SI vs SM (개발과 운영)

SI (System Integration):
새로운 시스템을 설계하고 구축하는 프로젝트성 업무 (건물 짓기)

SM (System Maintenance):
구축된 시스템을 안정적으로 운영하고 보수하는 업무 (건물 관리)

솔루션 (Solution)

특정 비즈니스 문제를 해결하기 위해 미리 만들어진 "완제품 소프트웨어 패키지"입니다. (예: ERP, 보안 솔루션)

프레임워크 (Framework)

개발을 효율적으로 하기 위해 미리 짜여진 "표준 뼈대(Kit)"입니다. 개발자는 이 뼈대 안에서 필요한 기능만 채워 넣습니다.

MVP (Minimum Viable Product)

고객의 피드백을 받기 위해 최소한의 핵심 기능만 담아 빠르게 출시한 제품입니다. 큰 프로젝트 전 '시장성 검증'을 위해 필수적입니다.

배포(Deployment): 개발 환경의 코드를 실제 서버에 올려 고객이 쓰게 만드는 것

트래픽(Traffic): 특정 시간대 시스템을 이용하는 사용자 데이터의 총량

로그(Log): 시스템에서 일어난 모든 일에 대한 디지털 기록(발자취)

AI Computing Architecture

왜 AI는 GPU를 원하는가:
순차 방식(CPU) vs 병렬 방식(GPU)

"인공지능 연산의 본질은 수억 번의 '행렬 곱셈'입니다.
이 거대한 계산 뭉치를 처리하는 방식의 차이가 성능을 결정합니다."

CPU

"소설책을 정독하는 독서가"

복잡한 논리나 명령어를 순서대로(Sequential) 하나씩 완벽하게 처리합니다. 하지만 단순 반복 계산이 수억 개라면 줄을 서서 기다려야 하는 병목 현상이 발생합니다.

GPU

"수학 문제를 동시에 푸는 수천 명"

수만 개의 픽셀을 동시에 칠하기 위해 탄생했습니다. 수천 개의 코어가 동시에(Parallel) 행렬 연산을 수행하므로, 학습 시간을 수천 배 이상 단축시킵니다.

AI 연산의 본질: Matrix

AI가 고양이를 인식할 때, 내부에서는 거대한 수치 데이터(가중치)들이 격자 형태의 행렬로 곱해집니다.

Conclusion:
CPU가 책 한 권을 처음부터 끝까지 읽는 동안,

엔비디아 (NVIDIA) : 지능의 공장

AI 연산의 뇌세포 역할을 하는 GPU 시장의 절대 강자입니다. 수조 개의 연산을 동시 처리하는 능력을 통해 AI 시대의 '표준 엔진'을 공급합니다.

HBM (고대역폭 메모리) : 지능의 고속도로

GPU가 아무리 빨라도 데이터 공급이 늦으면 무용지물입니다. HBM은 데이터를 수만 개의 통로로 쏟아붓는 '초고속 데이터 도로'입니다. SK하이닉스가 독보적인 이유입니다.

테슬라 (Tesla) : 행동의 신체

뇌(GPU+HBM)가 만든 지능을 현실의 움직임으로 바꾸는 기업입니다. 자율주행 데이터와 로봇 기술로 AI가 거주할 '가장 완벽한 신체'를 설계합니다.
